Module 01

Introduction to Python

What Python is, why it matters, and how to start writing it. The foundation everything else builds on.

What is Python?
Language overview & use cases

Python is a high-level, interpreted language with readable, English-like syntax. It's used across web development, data science, machine learning, and automation — and for platform work, automation is where it earns its keep most quickly. Repetitive tasks like pulling contract data, validating onboarding records, or hitting APIs don't need to be manual.

Python is "interpreted," meaning it runs line by line — you don't compile it first. That makes it fast to experiment with and easy to debug.

Worksuite Scenario

A client onboards 200 contractors at once. Instead of manually reviewing each record, a Python script processes every row of the import file, validates required fields against Worksuite's schema, and flags incomplete ones — missing tax forms, unsigned agreements, blank payment details.

Variables & Data Types
Storing and working with data

Variables are named containers for data your script needs to reference. Data types determine what you can do with a value. Python's core types: str (text), int (whole numbers), float (decimals), bool (True/False).

# Fields from a Worksuite API response — same types, same names
tenant_slug    = "meridianhealth"    # identifies the tenant
work_type      = "SOW"               # from /rest/v1/worktypes/
headcount      = 340                 # int, not "340" — math will break otherwise
is_enterprise  = True
Worksuite Scenario

The Worksuite API returns headcount as a number and is_enterprise as a boolean. Storing them at the top of your script means every downstream action — compliance checks, report generation, API calls — pulls from the same source without hardcoding values across 12 different places.

print() & Jupyter Notebooks
Output, feedback & your workspace

print() is your live feedback mechanism — it outputs text, variable values, or messages to the screen while your script runs. Jupyter Notebooks let you run code in chunks and see results immediately, which is ideal for exploratory analysis.

# Fields come directly from the /rest/v1/partners/{id}/ response
print(f"Processing {partner['display_name']}...")
print(f"Rate: ${partner['hourly_rate']} | Status: {partner['status']}")
Worksuite Scenario

When running a script that loops through partners from /rest/v1/partners/ to bulk-update compliance statuses, print statements tell you exactly where the script is — and exactly where it stopped if something goes wrong. Jupyter is ideal for digging into a client's onboarding completion rate before a quarterly business review.

Module 02

Python Basics

Conditionals, loops, and lists — the three tools you'll use in virtually every script you write.

Conditionals
if / elif / else

Conditionals branch your logic based on rules. Python uses if, elif (else-if), and else to evaluate conditions in order — the first one that's True runs, the rest are skipped.

if days_until_expiry <= 30:
    status = "Urgent — renewal required"
elif days_until_expiry <= 90:
    status = "Upcoming — begin renewal process"
else:
    status = "Active"
Worksuite Scenario

Every contractor contract in Worksuite gets a status without you reviewing each one manually. One block of logic handles the entire workforce roster in a single pass.

For Loops
Iterating over a collection

A for loop runs a block of code once for every item in a sequence. It's the go-to tool for processing a list of records, files, or API responses without repeating yourself.

# /rest/v1/partners/ returns paginated results under "results" key
for partner in response["results"]:
    if not partner.get("tax_form_on_file"):
        flagged.append(partner["display_name"])
    if not partner.get("payment_account_linked"):
        incomplete.append(partner["display_name"])
Worksuite Scenario

When an enterprise client onboards 200 contractors at once, a for loop iterates through every row of the import file, validates each record against Worksuite's required fields, and flags incomplete ones — instead of anyone reviewing 200 rows by hand.

While Loops
Repeating until a condition changes

A while loop keeps running as long as a condition is True. Use it when you don't know in advance how many times you'll need to repeat — like polling a status until it changes. Always make sure the condition will eventually become False, or the loop runs forever.

# Poll /rest/v2/payments/{id}/ until status transitions to "paid"
while payment["status"] != "paid":
    resp    = requests.get(f"{BASE_URL}/rest/v2/payments/{payment_id}/", headers=headers)
    payment = resp.json()
    time.sleep(30)
notify_team("Batch payment cleared")
Worksuite Scenario

When polling Worksuite's API to confirm a bulk payment batch has cleared, a while loop checks every 30 seconds and only stops — then fires a confirmation notification — once it returns a successful status.

Lists
Ordered, mutable collections

Lists store ordered collections of items that can grow, shrink, and change. Items are accessed by index (starting at 0). Key methods: append() adds to the end, remove() deletes by value, len() gives the count, and slicing (list[1:3]) extracts a range.

pending_approvals = ["CTR-801", "CTR-802", "CTR-803"]

pending_approvals.append("CTR-884")   # new contract in
pending_approvals.remove("CTR-801")   # approved, out
len(pending_approvals)                  # 3 — current queue
Worksuite Scenario

Open contractor approvals, pending compliance documents, flagged invoices — any queue that grows and shrinks over time lives naturally in a list. The length of the list is your real-time backlog count.

Module 03

Functions & Modules

How to write reusable code, organize it into logical blocks, and extend Python with external libraries.

Functions
Reusable blocks of logic (DRY principle)

Functions are reusable blocks of code. Define once, call as many times as needed — this is the DRY principle (Don't Repeat Yourself). They take parameters as inputs and use return to send output back. Variables defined inside a function are local — they don't exist or interfere outside of it (scope).

def utilization_rate(active_contractors, total_seats):
    return round((active_contractors / total_seats) * 100, 1)

# Use it anywhere — onboarding reports, QBR decks, dashboards
utilization_rate(178, 200)  # returns 89.0
Worksuite Scenario

When Worksuite changes how "active" is defined — say, to exclude contractors in offboarding status — you update one function, and every report that calls it is instantly correct. Without functions, you'd be hunting down the calculation across every script individually.

Classes
Blueprints for objects

Classes bundle data (attributes) and behavior (methods) into one structure. You define the blueprint once, then create as many instances as you need. __init__ is the constructor — it runs whenever you create a new instance.

# Attribute names mirror the /rest/v1/partners/{id}/ response body
class WorksuitePartner:
    def __init__(self, partner_id, display_name, hourly_rate, contract_end):
        self.id           = partner_id
        self.display_name = display_name
        self.hourly_rate  = hourly_rate
        self.contract_end = contract_end

    def days_until_expiry(self):
        return (self.contract_end - date.today()).days

p = WorksuitePartner("prt_a4k92x", "Priya Nair", 125.00, contract_end)
p.days_until_expiry()  # e.g. 42
Worksuite Scenario

Every contractor becomes an instance of the class — consistent structure across the entire engagement. Methods like is_compliant() or generate_invoice() live right alongside the data they need.

Built-ins & External Libraries
Python's toolbox, plus the community's

Built-in functions are always available — no imports. Common ones: len(), sorted(), max(), min(), sum(), type(). External libraries are installed via pip and massively extend what Python can do. Modules are Python files you import to add functionality.

# Built-ins — no import needed
sorted(payments, key=lambda p: p["amount"])
max(contract_values)

# External libraries — install with: pip install pandas requests
import pandas as pd
import requests

# Hit the Worksuite API directly — auth via Bearer token
resp     = requests.get("https://api.worksuite.com/rest/v1/partners/",
                        headers={"Authorization": f"Bearer {api_key}"})
partners = resp.json()["results"]
# Or load a Worksuite partner export CSV
df      = pd.read_csv("worksuite_partner_export.csv")
missing = df[df["compliance_status"].isna()]
Worksuite Scenario

pandas loads a client's 5,000-row contractor export, filters for workers with missing compliance documents, and produces a clean exceptions report in a few lines. requests lets you hit Worksuite's API directly — pulling real-time contract statuses or triggering workflows programmatically.

Module 04

Data Structures

Dictionaries, sets, and tuples — the right container for the right job. Plus sorting and searching at scale.

Dictionaries
Key-value pairs for structured records

Dictionaries store data as key: value pairs. Look up values by key — fast, readable, and flexible. Keys must be unique. You can add new keys, update values, or delete entries at any time.

# Shape mirrors a /rest/v1/partners/{id}/ response body
partner = {
    "id":           "prt_a4k92x",
    "display_name": "Priya Nair",
    "work_type":    "Independent Contractor",
    "hourly_rate":  125.00,
    "status":       "pending_compliance",
    "contract_end": "2026-06-30"
}

partner["status"]                          # "pending_compliance"
partner["hourly_rate"] = 135.00           # update rate
partner.get("notes", "none")              # safe default
Worksuite Scenario

Each contractor in a processing script becomes a dictionary. You look up contractor["compliance_status"] to decide whether to trigger an alert, or update the rate after an adjustment — without touching anything else in the record.

Sets & Tuples
Uniqueness and immutability

Sets are unordered collections with no duplicates — ideal for deduplication and membership checks. Tuples are like lists but immutable (can't be changed after creation) — use them for data that should never change mid-script.

# Sets — dedup partner IDs from two sources
gaps = set(worksuite_partner_ids) - set(payroll_system_ids)
# gaps = partners in Worksuite missing from payroll

# Tuples — fixed API config
api_config = ("https://api.worksuite.com", "/rest/v2/", api_key)
# Nothing in the script can accidentally overwrite this
Worksuite Scenario

Pulling contractor IDs from Worksuite and comparing them to a client's payroll system — one set subtraction instantly returns every worker missing from payroll, with no duplicate noise. Contractor permissions stored as a set make it structurally impossible for the same permission to appear twice.

Sorting & Searching
Efficiency at scale

sorted() returns a new sorted list; .sort() sorts in place. Linear search scans every item — fine for small lists, slow at scale. Binary search on a sorted list is dramatically faster (O(log n) vs O(n)), which matters when processing thousands of contractor records.

# Sort partners by health score, worst first
at_risk = sorted(partners, key=lambda p: p["health_score"])

# Sort payments by amount, largest first
payments.sort(key=lambda p: p["amount"], reverse=True)
Worksuite Scenario

Scanning through 50,000 Worksuite contractor records to find one by ID is slow. Binary search finds that record in a fraction of the time — the difference between a script that runs in two seconds and one that runs for twenty minutes.

Module 05

Error Handling & Debugging

Exceptions, try-except blocks, and the debugging toolkit that keeps your scripts reliable when real data shows up.

Common Exceptions
The errors you'll see most often

Exceptions are Python's way of flagging that something went wrong. They crash your program unless you handle them. Knowing the common types lets you anticipate and prevent them.

NameError    — used a variable before defining it
TypeError    — wrong type ("340" + 1 fails)
KeyError     — dict key doesn't exist
IndexError   — list index out of range
ValueError   — right type, wrong value (int("abc"))
AttributeError — method doesn't exist on the object
Worksuite Scenario

The Worksuite API occasionally returns numeric fields like hourly_rate as a string ("125.00") depending on the endpoint version. When you try hourly_rate * hours_worked, Python throws TypeError. Fix: float(partner["hourly_rate"]) * hours_worked.

Try-Except Blocks
Graceful error handling

Wrap risky code in a try block. If it fails, the except block catches it and lets you handle it gracefully. finally always runs — use it for cleanup. This is what separates a fragile script from a reliable one.

for partner in partners:
    try:
        process_payment(partner["id"])
    except KeyError as e:
        log_error(partner["display_name"], f"Missing field: {e}")
    except FileNotFoundError:
        log_error(partner["display_name"], "Export file not found")
    finally:
        close_connection()  # always runs
Worksuite Scenario

A nightly script processes contract exports for 30 clients. One client uploads a malformed file. Without error handling, the script crashes at client 7 and the other 23 never run. With try-except, you catch the failure, log it, and keep processing — then send a summary of exceptions at the end.

Debugging Strategies
Finding bugs systematically

Print debugging is the fastest first move — add print statements to inspect variable values at key points. The debugger lets you set breakpoints and step through code line by line. Read the traceback — it tells you the exact file, line number, and error type. Rubber duck debugging (explaining your code out loud) works surprisingly well for logic errors.

# Print debugging — inspect values at the key moment
print(f"rate: {partner['hourly_rate']}, hours: {hours}, total: {total}")

# Safe dict lookup — avoids KeyError crashes
tax_id = partner.get("tax_id", "Not provided")

# Check type when behavior is unexpected
print(type(partner["hourly_rate"]))  # str? float?
Worksuite Scenario

An invoice total calculation is producing wrong output for certain contractors. A single print line before the return statement immediately shows whether bad data is coming in or the formula itself has a bug. If the problem only appears on one of 200 records, a breakpoint and step-through will catch the exact moment a None slips in where a contract value was expected.

Module 06

Testing & Version Control

How to prove your code works, catch regressions before they matter, and never lose work again.

Unit Testing with pytest
Proving individual functions work correctly

Unit tests verify that individual functions produce correct output. pytest is the standard Python testing framework — write test functions that start with test_, use assert to check expected values, and run them all with pytest in the terminal. Fixtures set up reusable test data so you don't copy-paste the same setup into every test.

def test_utilization_rate():
    assert utilization_rate(178, 200) == 89.0
    assert utilization_rate(0, 200)   == 0.0
    assert utilization_rate(200, 200) == 100.0

# Fixture — reusable test partner matching /rest/v1/partners/{id}/ shape
@pytest.fixture
def sample_partner():
    return {"id": "prt_test01", "display_name": "Test User",
            "hourly_rate": 100.00, "status": "active"}
Worksuite Scenario

A compliance scoring formula works perfectly. Three months later, a colleague updates the logic for a new contractor classification Worksuite just added. A test written on day one immediately flags that the updated formula returns incorrect scores for legacy contract types — a bug that would have silently corrupted compliance reporting if it reached production.

Git & Version Control
Checkpoints, branches, and collaboration

Git tracks every change you make to code so you can revert mistakes, see history, and collaborate without overwriting each other. Commits are save points. Branches let you build features in isolation without affecting working code. GitHub hosts repositories in the cloud and adds pull requests and code review on top.

git init               # start tracking
git add filename.py    # stage specific file
git commit -m "msg"   # save snapshot
git log                # see full history
git diff               # see what changed
git checkout -b name  # new branch
git merge branchname  # merge changes in
Worksuite Scenario

You're refactoring a contract health scoring script the whole team uses. Something breaks. git log shows every change since the last working version; git diff shows exactly what changed. Branches mean one person can add a new Worksuite API integration while another fixes a bug — neither blocks the other, and neither breaks what's in production.
